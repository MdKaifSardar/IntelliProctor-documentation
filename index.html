IntelliProctor: A Next-Generation, Privacy-Preserving Edge-AI Proctoring Environment for Secure Online Technical Interviews

Author: Systems Security & AI Safety Researcher
Date: July 2026
Scope: Edge Computer Vision, Privacy-Preserving ML, Systems Security
Status: Peer-Review / Implementation Spec
Abstract

Distributed assessments present a high-stakes deployment environment where the incentives of the users directly oppose the operational objectives of the automated system. Traditional automated proctoring platforms resolve this tension through aggressive centralized video surveillance, transmitting raw biometric feeds to cloud servers, introducing severe privacy liabilities, demographic biases, and susceptibility to model manipulation. This paper presents IntelliProctor, an alternative, privacy-preserving, and robust edge-AI architecture for integrity verification. By combining a low-latency native desktop environment sandbox (Flutter) with local edge-inference models (MediaPipe and YOLOv8), IntelliProctor implements a zero-data-leakage verification model. We present the mathematical framework for dynamic 3D head pose estimation, a robust temporal risk aggregation engine that resists alert spam, and structural security measures (HMAC compiler signatures and Next.js server-side RBAC) that enforce system reliability. We frame this architecture as a practical case study in AI safety engineering, demonstrating that decentralized, localized machine learning mitigates systemic risks while ensuring operational alignment and candidate dignity.

1. Introduction & Ethical Alignment in AI

The rapid transition of academic and corporate evaluation pipelines to distributed remote environments has catalyzed the deployment of automated surveillance systems. In these high-stakes deployment scenarios, candidates are evaluated on technical competencies, while the system is tasked with verifying the integrity of the environment. In traditional setups, this verification relies heavily on cloud-based video audit systems.

From the perspective of AI Safety and Ethics, cloud-based automated surveillance systems present major failure modes:

  1. Biometric Data Leakage: Sending high-resolution video streams of candidates' private homes containing biometric, demographic, and environmental data to remote servers exposes them to massive security breaches.
  2. Lack of Agency: Candidates are forced to agree to invasive data capture, with no guarantees about how their raw recordings are stored, audited, or repurposed.
  3. Demographic and Algorithmic Bias: Deep neural networks trained on non-representative datasets exhibit higher error rates and false-positive rates when evaluating candidates with diverse skin tones, physical abilities, or facial structures.
  4. Black-Box Automation: Disqualifying a candidate based on brief anomalies (e.g., looking away to read a formula, ambient room noise, or a family member walking by) without human audit loops violates basic principles of fairness and safety.

IntelliProctor challenges the paradigm of cloud-based surveillance. Our core thesis is that AI Safety is a systems-level property. We design a secure, low-latency, and privacy-preserving assessment loop by shifting machine learning inference to the edge and incorporating dynamic, personalized calibration.

2. Core System Architecture & IPC Guardrails

IntelliProctor employs a hybrid, split-stack architecture to separate administrative task coordination from localized environment validation. The stack consists of three major components:

Figure 2.1: Technical diagram representing the IntelliProctor split-stack architecture, detailing direct telemetry pipelines bypassing student runtime environments.

The architecture enforces specific safety and security invariants at each boundary:

A. Flutter-Managed Native Sandbox

The desktop client, written in Flutter, serves as a secure wrapper for candidate assessments. It enforces true fullscreen mode with low-level platform hooks to intercept windows keyboard shortcuts (e.g., Alt+Tab, Windows keys) and masks desktop panels. When initialized, it launches the Python AI sidecar subprocess locally through a utility called BackendLauncher. The sandbox acts as a physical security barrier, ensuring the candidate cannot easily manipulate visual sensors or access external help tools.

B. FastAPI WebSockets Telemetry Loop

Communication between the secure Flutter screen and the Python AI Engine occurs entirely over local JSON-based WebSockets (ws://localhost:8000/ws/video and /ws/commands). This design provides two key benefits:

  • Low-Latency Real-Time Telemetry: Frames are base64-encoded, processed locally at 30 FPS, and returned to the UI with annotated bounding boxes and yaw/pitch tracking indicators within a 33ms window.
  • Zero Network Footprint: Real-time processing remains purely local. No raw video feed is transmitted over the internet, rendering intercept attacks on sensitive video feeds impossible.

C. Direct-to-Firestore Telemetry Writing (Integrity Bypass)

A common vulnerability in student-side security tools is the manipulation of the frontend environment. If a candidate injects code into the client-side JavaScript or Flutter context, they can intercept calls and drop "cheat" alerts before they reach the recruiter.

Adversarial Telemetry Mitigation

To prevent candidate manipulation of the frontend client, IntelliProctor implements a direct logging bypass. The local Python AI engine, running as an isolated system process, computes risk scores and communicates directly with Cloud Firestore using encrypted, short-lived tokens. The candidate's UI is bypassed entirely for report transmission, preventing client-side code modification or memory injection from tampering with the computed integrity history.

3. Local Computer Vision & Gaze Estimation Mathematics

The core of IntelliProctor's gaze tracking is a 3D Pose Solver that evaluates the angular deflection of a candidate's face in physical space relative to the camera sensor. Rather than using generic face detection heuristics, the platform utilizes Google's MediaPipe FaceLandmarker model (face_landmarker.task), a lightweight convolutional neural network optimized for real-time mobile and edge inference. The model outputs a set of 478 3D landmarks (consisting of the standard 468 facial mesh points plus 10 iris tracking landmarks). Gaze estimation is implemented via a dual-pipeline architecture cross-calibrated between the browser client-side engine and the native desktop sidecar.

A. Perspective-n-Point (PnP) Solver (Python Engine)

In the desktop sandbox backend (backend/app/detectors/face_detector.py), the engine extracts a specific subset of six 2D landmarks from the MediaPipe Face Mesh coordinates to form a robust facial structure:

  • Left Eye Outer Corner: Landmark 33
  • Right Eye Outer Corner: Landmark 263
  • Nose Tip (Pronasale): Landmark 1
  • Left Mouth Corner (Cheilion): Landmark 61
  • Right Mouth Corner (Cheilion): Landmark 291
  • Chin Bottom-Middle (Menton): Landmark 199

These 2D coordinates are aligned against a static, canonical 3D face model centered at the nose tip:

P_{3D} = \begin{bmatrix} [-225.0, -170.0, 135.0] & \text{(Left Eye Corner)} \\ [225.0, -170.0, 135.0] & \text{(Right Eye Corner)} \\ [0.0, 0.0, 0.0] & \text{(Nose Tip)} \\ [-150.0, 150.0, 125.0] & \text{(Left Mouth Corner)} \\ [150.0, 150.0, 125.0] & \text{(Right Mouth Corner)} \\ [0.0, 330.0, 65.0] & \text{(Chin)} \end{bmatrix}

Given camera focal length f \approx w_{img} and optical center coordinates (c_x, c_y) \approx (w/2, h/2), we construct the camera intrinsic matrix K:

K = \begin{bmatrix} f & 0 & c_x \\ 0 & f & c_y \\ 0 & 0 & 1 \end{bmatrix}

The solver executes the Perspective-n-Point routine (cv2.solvePnP) to compute the translation vector T and the rotation vector R_{vec} that minimizes the projection error of the 3D model landmarks onto the 2D image plane coordinates:

s \begin{bmatrix} u \\ v \\ 1 \end{bmatrix} = K \Big( R \cdot P_{3D} + T \Big)

The rotation vector R_{vec} is converted to a 3x3 rotation matrix R \in \text{SO}(3) using the Rodrigues transform (cv2.Rodrigues(rot_vec)). We apply RQ Decomposition (cv2.RQDecomp3x3) to decompose R into Euler angles (Yaw, Pitch, Roll), which are normalized to represent gaze shifts:

Yaw_{Normalized} = \frac{\theta_{yaw}}{90.0}, \quad Pitch_{Normalized} = \frac{\theta_{pitch}}{90.0}

B. Matrix-Based Procrustes Alignment (Browser Client-Side Engine)

In the web console (frontend-web/src/lib/proctoring/MediaPipeEngine.ts), the browser loads the complete FaceLandmarker model and configures outputFacialTransformationMatrixes: true. MediaPipe calculates this transformation matrix internally by fitting the canonical face model to all predicted 3D landmarks using Orthogonal Procrustes Analysis. This solves for the rigid transformation mapping the canonical coordinates to the camera coordinate space:

P_{camera} = R_{3\times 3} \cdot P_{canonical} + T_{3\times 1}

The result is returned as a 4x4 matrix M in a column-major Float32Array of length 16:

M = \begin{bmatrix} r_{00} & r_{01} & r_{02} & t_x \\ r_{10} & r_{11} & r_{12} & t_y \\ r_{20} & r_{21} & r_{22} & t_z \\ 0 & 0 & 0 & 1 \end{bmatrix}

Because the array is column-major, the Z-axis orientation vectors (rotation elements r_{20}, r_{21}, r_{22}) are mapped to indices 2, 6, and 10. The web engine extracts these elements and derives the Yaw and Pitch angles analytically using Tait-Bryan convention: 

const m = result.facialTransformationMatrixes[0].data; // Float32Array (16)
const r20 = m[2];  // Column 0, Row 2
const r21 = m[6];  // Column 1, Row 2
const r22 = m[10]; // Column 2, Row 2

const pitch = Math.asin(-r21);
const yaw = Math.atan2(r20, r22);

By cross-calibrating the scaling scales between the Python PnP solver and the browser Procrustes matrix analyzer, both stacks evaluate candidate orientation against the same recruiter-configured angular thresholds (e.g. Yaw threshold: 0.20 rad, Pitch Up threshold: 0.20 rad, Pitch Down threshold: 0.15 rad) uniformly.

4. Robust Temporal Risk Aggregation & Cooldown

A primary failure mode in AI-driven enforcement is over-sensitivity. Raw predictions from deep models (like YOLOv8 or MediaPipe landmark detections) are noisy. A single frame where the candidate's eyes look at the sidebar or the model misclassifies a coffee mug as a "cell phone" should not trigger a high-severity alert. If an algorithm is too sensitive, candidates experience severe anxiety, causing them to perform poorly, and recruiters are flooded with thousands of false alerts.

IntelliProctor implements a Temporal Aggregator and Cooldown Risk Engine in backend/app/analysis/behavior.py and risk_engine.py:

A. Temporal Violation Buffering

Instead of evaluating single-frame signals, the BehaviorAnalyzer maintains rolling counters for consecutive frames of anomalies. A signal is emitted only if it persists over a specified duration:

  • Face Missing: Requires N_{frames} > 30 (~1.0s at 30 FPS) to trigger FACE_NOT_VISIBLE.
  • Gaze Aversion: Requires N_{frames} > 3 (~0.1s) to trigger LOOKING_AWAY.
  • Pitch Violation: Requires N_{frames} > 5 (~0.15s) to trigger PITCH_VIOLATION.

B. Weighted Risk Engine & Alert Cooldowns

When signals are verified, they are mapped to specific risk severities and scores. The RiskEngine sums active severity weights, applying a 2-second alert cooldown window to prevent flooding.

5. Privacy-Preserving AI & Zero-Data-Leakage Design

In high-stakes distributed assessments, protecting the candidate's privacy is a fundamental safety requirement. Biometric data, particularly facial structures and voice recordings, are classified as highly sensitive under data protection laws (e.g., GDPR, CCPA). If a proctoring system uploads video recordings to cloud servers, it creates massive target directories for malicious actors.

IntelliProctor addresses this liability through a Zero-Data-Leakage Architectural Principle:

Architectural Privacy Guarantee

No raw frame, camera stream, or audio sample is ever written to disk or transmitted across local borders. All computer vision processing occurs entirely within the candidate's local CPU/GPU registers. When an anomaly is detected, the raw frame is analyzed, the statistical metric is calculated, and the frame is immediately dropped from RAM. The only data sent to Firestore is the anonymous, structural telemetry metadata detailing the anomaly type and timestamp.

6. Ethical AI, Calibration, and Demographic Fairness

A major flaw in vision-based detection systems is their susceptibility to demographic bias. Standard facial tracking architectures can exhibit higher error rates or struggle to lock on when evaluating candidates with diverse skin tones, deep facial features, or those wearing glasses or religious headwear. Furthermore, candidates may sit at different heights, work in rooms with uneven lighting, or position their camera at steep angles.

If an AI proctoring tool uses absolute, rigid thresholds to evaluate gaze, it will consistently generate false positives for candidates whose natural alignment differs from the model's training baseline.

IntelliProctor mitigates these systemic biases using an Active Baseline Calibration System (backend/app/analysis/gaze_calibrator.py):

A. Baseline Posture Normalization

Before starting an exam, the platform initiates a calibration stage. The candidate is prompted to sit comfortably and look directly at the center of their screen. The GazeCalibrator collects 60 frames of raw facial coordinates. It validates that the user is looking at the screen (e.g., confirming face presence and tracking consistency) and calculates baseline values:

\mu_{yaw} = \frac{1}{60} \sum_{i=1}^{60} Yaw_{i}^{raw}, \quad \mu_{pitch} = \frac{1}{60} \sum_{i=1}^{60} Pitch_{i}^{raw}

For the remainder of the assessment, all yaw and pitch values are calculated relative to this baseline:

Yaw_{calibrated} = Yaw^{raw} - \mu_{yaw}, \quad Pitch_{calibrated} = Pitch^{raw} - \mu_{pitch}

This normalizes the tracking coordinate frame to each user's unique setup. It compensates for camera angle offsets, lighting environments, and structural facial differences, preventing demographic or postural factors from triggering false-positive alerts.

B. Environmental Noise Calibration

Similarly, candidates live in diverse socioeconomic environments. A candidate taking an assessment in a quiet suburban home has a lower ambient noise level than one taking it in a dense urban apartment with street traffic. If a static noise threshold is applied, the latter will be constantly flagged for audio violations. IntelliProctor allows recruiters to customize the acoustic threshold or calibrate the microphone dynamically to establish a baseline noise ceiling, ensuring fair evaluations regardless of background environment.

7. Security & Cryptographic Guardrails

An AI safety framework must secure the execution environment to ensure data integrity and prevent candidate manipulation of test results. IntelliProctor implements several structural security measures:

A. HMAC SHA-256 Signed Compiler Callbacks

When a candidate submits code within the built-in IDE (frontend-web/src/app/candidate/), the system compiles and executes the code via Judge0. To prevent candidates from spoofing execution results (e.g., sending fake positive test case responses), the callback API validates the authenticity of the compiler's response.

The Next.js callback receiver (frontend-web/src/app/api/interviews/callback/route.ts) requires a signature computed using HMAC SHA-256 with a shared secret: 

import { createHmac, timingSafeEqual } from 'crypto';

export function verifySignature(payload: string, signature: string, secret: string): boolean {
  const computed = createHmac('sha256', secret)
    .update(payload)
    .digest('hex');
    
  return timingSafeEqual(
    Buffer.from(computed, 'hex'),
    Buffer.from(signature, 'hex')
  );
}

By enforcing timingSafeEqual comparisons, the system prevents timing-attack vectors, ensuring that code evaluations cannot be spoofed.

B. Server-Side Firestore Security

Unlike typical Firebase applications that initialize the Firestore client SDK directly in browser components, the IntelliProctor Web Console relies entirely on Next.js Server Actions ("use server") and the Firebase Admin SDK.

Because the client-side environment contains no Firestore database keys or direct access methods, candidates cannot execute arbitrary reads/writes or query other candidates' records, ensuring strict data isolation at the API level.
