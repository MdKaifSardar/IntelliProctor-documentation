IntelliProctor Architecture Manual

A comprehensive engineering resource guide for Next-Generation Online Technical Interview Proctoring

System Overview

IntelliProctor is a security-focused, privacy-preserving assessment ecosystem designed for high-stakes technical interviews. Unlike standard cloud-based proctoring platforms that rely on high-bandwidth video upload streaming, IntelliProctor runs edge-based local machine learning models directly on the candidate's device, ensuring user privacy and low latency while executing strict client sandbox constraints.

The ecosystem comprises three primary components:

  1. Next.js Management Console (Web Dashboard): Used by recruitment coordinators to schedule tests, organize pipelines, review security alerts, manage questions, and trigger AI-assisted generation workflows.
  2. Flutter Desktop Wrapper (Candidate Sandbox): A compiled native application wrapping the assessment console, locking keyboard inputs (Alt+Tab, Windows Key shortcuts), detecting window focus shifts, and orchestrating the Python process lifecycle.
  3. Python FastAPI Engine (Edge-AI Brain): A lightweight sidecar process running locally alongside the Flutter app. It handles high-frequency camera processing (MediaPipe Face Mesh and YOLOv8 object detection) and microphone voice activity verification over low-latency WebSockets.

Candidate Pipelines & Registry Operations

Recruiters invite candidates to assessment workflows by loading them into the database registry. The registry enforces strict schemas and state transitions to prevent unauthorized joins or memory leakage.

A. Zod-Validated Bulk CSV Registry Imports

When recruiters import rosters via CSV, the Next.js API parses row records and validates email, name, and candidate attributes using Zod prior to database writing:

CSV Parser Schema Validation
import { z } from 'zod';

export const CandidateSchema = z.object({
  email: z.string().email(),
  name: z.string().min(2),
  role: z.string().optional(),
  assessmentId: z.string()
});

B. Candidate Lifecycle State Invariants

Each candidate profile transitions through a deterministic pipeline state machine:

Pipeline State Transition Cause Description / Security Level Granted
INVITED Recruiter adds email to registry Candidate record exists, link is not yet joinable.
JOINABLE Assessment schedule window opens Candidate is authorized to log in and start the calibration sequence.
IN_PROGRESS Calibration completes, assessment launches Telemetry recording starts. Lockouts active.
COMPLETED Timer expires or candidate manual submission Telemetry locks, preventing subsequent writes to Firestore.

Question Architect

Technical questions are managed using a Group-First model inside Firestore. In this pattern, questions do not float as independent root records; instead, they are child subcollections of a primary parent Question Group.

This model offers two critical benefits:

  • Difficulty & Description Categorization: The parent Group defines overall metadata (e.g. difficulty: easy | medium | hard, techStack: react | python). Child questions inherit these attributes.
  • Randomization Blueprints: When creating an assessment test, recruiters define a blueprint referencing Group IDs. The test generator randomly selects K questions from the respective Groups at start time.
Structural Database Invariant: Cascading Deletions

Because Firestore subcollections are not automatically cleaned up when parent documents are deleted, deleting a Group would normally leave "ghost" questions orphaned in the database. IntelliProctor enforces cascading deletions via server-side operations (using the Firebase Admin SDK inside Next.js Server Actions), cleaning up all child question documents recursively when their parent Group document is deleted.

AI Interview Copilot (Interview Generator Assistant)

To automate assessment drafting, IntelliProctor embeds an AI Interview Generator directly into the dashboard. When recruiters enter parameters (e.g. tech stack, target role, topics, and difficulty constraints), a generative AI model drafts programming questions, coding starter code, test suites, and expected outputs.

Flutter Native Sandbox (Desktop Client)

To isolate the execution environment, candidates run exams inside a locked desktop sandbox written in Flutter. The client wraps the browser console and secures the environment at the OS level:

  • Full-Screen Lockouts: Enforces true fullscreen display mode, preventing candidates from minimizing the application window.
  • Low-Level Shortcut Interceptions: Uses Windows native hooks (Win32 API) to override common task-switching keyboard events, blocking shortcuts such as Alt+Tab, Windows Key, and Ctrl+Esc.
  • Focus Loss Alerts: Triggers a high-priority risk score penalty if the sandbox window loses focus to an background process.
  • Process Lifecycle Manager: Automatically spawns the Python FastAPI edge-AI sidecar subprocess (BackendLauncher) when the sandbox launches, and kills it when the assessment closes, preventing background daemon leaks.

Web-Based CV Proctoring & Face/Gaze Estimation (WASM Modules)

For candidates using the web-based console, IntelliProctor runs a zero-dependency, lightweight in-browser proctoring module using WebAssembly:

  • MediaPipe FaceLandmarker Integration: Loads the face landmarker binary from jsdelivr CDN. Computes 3D gaze mesh vectors by extracting rotations from the facialTransformationMatrixes array directly. This maps to the column-major indices to calculate Pitch and Yaw angles analytically using Tait-Bryan convention: 
    const pitch = Math.asin(-m[6]);
const yaw = Math.atan2(m[2], m[10]);
    This allows browser-based systems to estimate head pose coordinates in real time without importing large C++ OpenCV dependencies, utilizing WebAssembly (WASM) and WebGL pipelines.
  • Active Baseline Normalization (Calibration): Before launching the exam, candidates undergo a 60-frame baseline calibration. This calculates mean resting angles (\mu_{yaw}, \mu_{pitch}) to calibrate offsets: 
    Yaw_calibrated = Yaw_raw - μ_yaw;
    This compensates for physical webcam angles, sitting height, and facial geometry variations, eliminating biometric bias.
  • Tab Focus Monitoring: Tracks window state changes using the browser's Visibility API and window.blur listeners. Any tab-switching triggers immediate warnings and logs a TAB_BLUR telemetry record.
  • Ambient Sound Monitor: Utilizes the Web Audio API to calculate ambient microphone volume. Analyzes decibel RMS values relative to the calibrated background ceiling to flag whispering or talking.

Edge-AI Telemetry Brain (Python Backend & cv2.solvePnP)

The local Python AI engine acts as the primary evaluation brain, running two models in parallel:

  1. MediaPipe Face Mesh (6-Point PnP Head Pose): Extracts face landmarks in real-time. Instead of a full Procrustes fit, it tracks 6 landmarks (Chin, outer eyes, nose tip, mouth corners) and solves the Perspective-n-Point equations to monitor Yaw/Pitch deviation:
    s * [u, v, 1]T = K * ( R * P3d + T )
    Where K is the camera matrix, R is the 3x3 rotation matrix derived via the Rodrigues transform, and T is the translation vector. We decompose R using RQ Decomposition (cv2.RQDecomp3x3) to isolate the Euler yaw and pitch angles.
  2. YOLOv8 forbidden object detection: Evaluates frames at a lower frequency (e.g. 5 FPS) to identify prohibited items, specifically looking for classes like cell phone, laptop, headphone, or book.

To prevent false-positive spikes due to frame noise, the backend processes raw labels through rolling temporal buffers (e.g. a face must be missing for 30 consecutive frames before registering a FACE_NOT_VISIBLE violation). Verified violations are transmitted directly to Cloud Firestore using secure, write-only database keys, bypassing the candidate's browser runtime.

Secured Code IDE & Judge0 Compiler

The candidate console features a custom, multi-tab IDE with syntax highlighting. When a candidate submits code, the system validates the output securely:

  • Judge0 Remote Execution: Code is compiled and run against hidden test cases inside isolated sandbox containers managed by Judge0.
  • HMAC SHA-256 Callback Verification: To prevent candidates from intercepting the compiler's JSON responses and altering their test scores (e.g., changing failing tests to passing), Judge0 signs payload callbacks with a secret key using HMAC SHA-256. The Next.js API verifies this signature before recording grade updates.
  • Automatic Work Recovery: The IDE caches candidates' starter code in LocalStorage. If the browser crashes, the state recovers automatically when reloading the exam.

Firestore Rules Schema & Access Security

      

        To protect data integrity, the system implements a strict role-based authorization model. Administrative reads and writes (managing groups, questions, configurations) bypass client access rules entirely by running on Next.js Server Actions using the Firebase Admin SDK. Candidate access, on the other hand, is restricted via granular rules:
      

      

        
Firebase Security Rules

        
service cloud.firestore {
  match /databases/{database}/documents {
    // Only Admin Server Actions can manage assessment questions
    match /questionGroups/{groupId} {
      allow read: if request.auth != null;
      allow write: if false; // Denied to all client SDKs
    }
    
    // Candidates can write to their own telemetry, but cannot read or edit existing records
    match /assessments/{assessmentId}/telemetry/{logId} {
      allow create: if request.auth != null && request.auth.uid == resource.data.candidateId;
      allow read, update, delete: if false;
    }
  }
}
